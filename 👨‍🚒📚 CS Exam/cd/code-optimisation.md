# Code Optimisation

#### AST vs DAG vs CFG Graphs

**Directed Acyclic Graph (DAG) ⭐**

* Used for **code optimization** inside a **basic block**
* Graph with **directed edges** and **no cycles**
* Nodes represent variables, constants, or operations
* Same subexpression → **single node**
* Helps in: ⭐
  * <mark>Common subexpression elimination</mark>
  * <mark>Dead code elimination</mark>
  * Code motion
* Scope: **local optimization only**

**Abstract Syntax Tree (AST)**

* Tree representation of **program syntax**
* Generated after parsing
* Interior nodes → operators / constructs
* Leaf nodes → identifiers / constants
* Removes unnecessary grammar symbols (unlike parse tree)
* Used for: ⭐
  * <mark>Syntax analysis</mark>
  * <mark>Semantic analysis</mark>
  * Intermediate code generation
* Structure: **tree (hierarchical)**

**Control Flow Graph (CFG) ⭐**

* Graph representing **flow of control**
* Nodes → basic blocks
* Edges → possible control transfers
* Used for:
  * Loop detection
  * Reachability analysis
  * Data flow analysis
  * Global optimization
* May contain **cycles** (loops)

**Key Differences (Exam-Oriented)**

| Feature      | DAG                      | AST            | CFG              |
| ------------ | ------------------------ | -------------- | ---------------- |
| Structure    | Graph                    | <mark>Tree</mark>       | Graph            |
| Cycles       | Not allowed              | Not allowed    | <mark>Allowed</mark>      |
| Level        | Expression / Basic block | Program syntax | Program flow     |
| Optimization | <mark>Yes (local)</mark>          | No             | <mark>Yes (global)</mark> |
| Focus        | <mark>Computation</mark>          | Syntax         | Control          |

**One-Line Memory Hooks**

* **AST** → how code is written → for <mark>meaning</mark>
* **DAG** → how expressions are optimized → for <mark>efficiency</mark>
* **CFG** → how control flows → for <mark>execution flow</mark>

#### Nodes and Edges in DAG

<mark>Directed Acyclic Graph (DAG)</mark> is used to <mark>represent **basic blocks**</mark> in compiler optimization to <mark>eliminate redundant computations.</mark>

**Nodes**

Each node represents:

* A variable (leaf node)
* A constant (leaf node)
* An operator with operands (`+`, `-`, `*`, `/`, etc.) So,
* <mark>**Number of nodes</mark> = <mark>No. of distinct variables</mark> + <mark>No. of constants</mark> + <mark>No. of distinct operations**</mark>
* Common subexpressions share the **same node**

**Edges**

Edges represent **data dependency**:

* **Directed** from <mark>operand → operator</mark>
* If an operator has `k` operands, it contributes `k` incoming edges So,
* <mark>**Number of edges</mark> = <mark>sum of operands of all operator nodes</mark>**
* Unary operator → 1 edge
* Binary operator → 2 edges

**Key Point**

DAG minimizes nodes and edges by:

* Merging identical subexpressions
* Avoiding recomputation

**Example**

```
a = b + c
d = b + c
```

* Nodes: `b`, `c`, `(b+c)`, `a`, `d` → **5 nodes**
* Edges: `b → (b+c)`, `c → (b+c)`, `(b+c) → a`, `(b+c) → d` → **4 edges**

```
      b     c
       \   /
        \ /
       ( + )
       /   \
      a     d
```

Best view: **Fewer nodes and edges = more optimized code**

#### Data Flow Analysis & Code Optimisation

**Data Flow Analysis (DFA) ⭐**

* Technique to <mark>collect information about **program variables and expressions**</mark> at different program points
* Performed on <mark>**Control Flow Graph (CFG)**</mark>
* Foundation for **global code optimization**
* Based on **iterative equations** until fixed point

**Basic Terms**

* **Basic Block**: maximal sequence of statements with single entry and exit
* **CFG Node**: basic block
* **CFG Edge**: possible control transfer
* <mark>**`IN[B]`**: information before block B</mark>
* <mark>**`OUT[B]`**: information after block B</mark>
* **`GEN[B]`**: information generated by block B
* **`KILL[B]`**: information invalidated by block B

**Direction of Analysis**

* <mark>**Forward</mark> Analysis**: information <mark>flows along control flow</mark>\
  Example: Available Expression
* <mark>**Backward</mark> Analysis**: information <mark>flows opposite to control flow</mark>\
  Example: Live Variable

**Data Flow Equation (General)**

*   Forward:

    ```
    IN[B]  = ⋂ / ⋃ OUT[P]  for all predecessors P
    OUT[B] = GEN[B] ∪ (IN[B] − KILL[B])
    ```
*   Backward:

    ```
    OUT[B] = ⋂ / ⋃ IN[S]  for all successors S
    IN[B]  = GEN[B] ∪ (OUT[B] − KILL[B])
    ```

**Meet Operator**

* **Union (⋃)**: may analysis
* **Intersection (⋂)**: must analysis

**A. <mark>Live Variable Analysis**</mark> ⭐

* <mark>Variable is **live**</mark> if its <mark>value may be used later</mark>
* Type: <mark>**Backward analysis**</mark>
* Meet: <mark>**Union**</mark>
* GEN: variables used before definition
* KILL: variables defined
* Used for: <mark>**Dead Code Elimination**</mark>

**B. Available Expression Analysis** ⭐

* <mark>Expression is available</mark> if <mark>already computed and not killed
* Type: <mark>**Forward analysis**</mark>
* Meet: <mark>**Intersection**</mark>
* GEN: expressions computed
* KILL: operands redefined
* Used for: <mark>**Common Sub-expression Elimination**</mark>

**Code Optimization**

* Process of improving code **without changing semantics**
* Goal: reduce execution time, memory, power
* Performed on **Intermediate Representation (IR)**

**Types of Optimization**

1. **Machine Independent**
2. **Machine Dependent**

**A. Local Optimization**

* Applied within a **basic block**
* Techniques:
  * <mark>Common sub-expression elimination</mark> (via DAG)
  * <mark>Constant folding</mark>
  * <mark>Dead code elimination</mark>

**B. Global Optimization**

* Applied across **multiple basic blocks**
* Requires **Data Flow Analysis**
* Techniques:
  * <mark>Live variable based</mark> <mark>dead code elimination</mark>
  * Global sub-expression elimination
  * <mark>Code motion</mark> ⭐

**C. Loop Optimization**

* Improves frequently executed loops
* Techniques:
  1. <mark>**Loop Invariant Code Motion**</mark> ⭐
  2. <mark>**Strength Reduction**</mark> ⭐ Example: `i * 2 → i + i` ⭐
  3. <mark>**Induction Variable Elimination**</mark>

**Techniques:** ⭐⭐

1. **Dead Code Elimination** – Removes statements whose results are never used (based on live variable analysis)
2. **Common Sub-Expression Elimination** – Computes an expression once and reuses its value instead of recomputing
3. **Loop Invariant Code Motion** – Moves computations that do not change inside a loop to outside the loop
4. **Strength Reduction** – Replaces expensive operations with cheaper ones (e.g., `*` → `+`)
5. **Induction Variable Elimination** – Removes extra loop variables derivable from a primary induction variable
6. **Code Motion** – <mark>Relocates computations to points</mark> <mark>where they execute fewer times</mark> without changing semantics
7. **Constant Folding** – <mark>Evaluates constant expressions at compile time</mark>
8. **Constant Propagation** – <mark>Replaces variables with known constant values</mark>
9. **Copy Propagation** – <mark>Replaces variables that are simple copies with the original variable</mark>
10. **Algebraic Simplification** – Uses algebraic identities to simplify expressions
11. **Unreachable Code Elimination** – Removes code that can never be executed
12. **Peephole Optimization** – Performs small local optimizations on a short sequence of instructions

**GATE Focus Points**

* Direction of analysis
* Meet operator choice
* GEN/KILL construction
* Mapping:\
  DFA → enables optimization\
  Optimization → improves performance

**One-Line Memory Rule**

* **CFG + DFA = Global Optimization**
